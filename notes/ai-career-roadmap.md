# AI 学习路线（2026 修订版）

## 北极星目标

目标不是“会调用几个模型”，而是在毕业时成为一名**有扎实基础、能把 AI
系统做出来并能解释其限制的应用 AI 工程师**。这条路能够通向更前沿的
LLM、深度学习和研究工程工作。

“前沿 AI 研究员”通常还要求长期的数学深度、论文复现能力和研究经历；
这是毕业后的延伸方向，而不是本学期必须完成的承诺。当前最正确的策略是
先成为一名可靠的工程型学习者，再逐步向研究和前沿系统能力扩展。

## 当前状态（2026 S2）

- 已完成：开发环境、Git/GitHub、计算器、猜数字游戏、`for` 循环和九九
  乘法表。
- 正在学：Python 函数。
- 本学期课程：COMP9001、COMP9120、COMP5310、INFO5990。
- 设备：MacBook Air M2；本地开发足够。深度学习训练优先使用 Colab、Kaggle
  或课程资源，不急于购买 PC/GPU。

## 本学期的优先级

课程成绩和可迁移能力优先于“追最新模型”。学习内容应尽量服务于真实课程
作业，并沉淀为可展示的仓库成果。

| 课程 | 主能力 | 需要留下的证据 |
| --- | --- | --- |
| COMP9001 | Python、算法思维、测试和调试 | 小练习、独立复现、清楚的提交记录 |
| COMP9120 | SQL、数据库设计、查询优化 | ER 图、SQL 查询集、小型 PostgreSQL 项目 |
| COMP5310 | 数据清洗、可视化、统计推断、ML 管线 | 一个可复现的 EDA + 基线模型项目 |
| INFO5990 | 伦理、安全、数据质量、沟通和项目实践 | 项目 README、风险与限制说明、简短展示 |

### 每周节奏（正常教学周）

- 课程作业与复习：优先完成，并把可公开部分整理进仓库。
- 额外 6–8 小时：2 小时 Python/软件工程、2 小时数据或 SQL、1–2 小时数学、
  1–2 小时项目整理与复盘。
- 考核周：只做课程和维护，不追新框架。
- 每周日：15 分钟写“本周学会了什么、哪里卡住、下周唯一重点”。

## 分阶段路线

### 阶段 A：编程与数据基础（现在至本学期中段）

1. **Python**：函数、列表、字典、文件、异常、模块、基础 OOP。
2. **软件工程习惯**：可读命名、格式化、调试、最小测试、`README`、Git。
3. **SQL**：`SELECT`、过滤、聚合、`JOIN`、子查询、表设计、索引和查询计划。
4. **数据处理**：NumPy、Pandas、可视化、缺失值和异常值。
5. **数学并行开始**：代数与函数、向量/矩阵、概率、描述统计和导数的直觉。

完成标准：能独立读入真实 CSV，提出数据问题，清洗、分析、可视化并写出
结论；能写 SQL 解决相同的数据问题；能解释代码而不是只让它运行。

### 阶段 B：经典机器学习（本学期后段）

1. 问题定义、训练/验证/测试划分、数据泄漏。
2. 线性回归、逻辑回归、树、随机森林与一个聚类方法。
3. 指标选择、混淆矩阵、交叉验证、过拟合、误差分析。
4. 用 scikit-learn 完成一个端到端项目；重点是可复现和解释，不是 Kaggle 排名。

完成标准：能说明“这个模型为什么可信、哪里不可信、下一步应收集什么数据”。

### 阶段 C：深度学习与 LLM 基础（本学期后或假期）

在完成一个基础 sklearn 项目、理解矩阵和训练循环后开始，不必等到“所有
前置知识完美”。

1. PyTorch：Tensor、Dataset/DataLoader、模型、损失、反向传播、优化、保存/加载。
2. 深度学习：MLP、CNN、训练诊断、正则化、迁移学习。
3. Transformer：token、embedding、attention、encoder/decoder、推理成本。
4. LLM 工程：提示与结构化输出、embedding、检索增强生成（RAG）、评测、
   安全边界、成本/延迟权衡。
5. 微调只在能够明确说明数据、指标、基线和风险时再做；不要把“调用 API”
   误当成理解模型。

推荐顺序：先完成 [PyTorch Learn the Basics](https://docs.pytorch.org/tutorials/beginner/basics/intro.html)，再学习
[Hugging Face LLM Course](https://huggingface.co/learn/llm-course/chapter1/1)。

### 阶段 D：工程化、研究复现与求职作品集（持续进行）

1. **工程化**：Python 项目结构、依赖管理、测试、日志、配置、API、Docker、
   GitHub Actions、部署和监控。
2. **实验质量**：固定随机种子、数据版本、实验记录、基线、消融、错误分析。
3. **前沿能力**：阅读论文摘要和图表，复现一个小实验，比较不同模型/提示/检索
   策略，并用评测结果做决策。
4. **沟通与责任**：数据隐私、偏差、版权、安全、失败模式、成本和用户影响。

完成标准：有 3 个质量胜过数量的项目，每个都能安装、运行、解释、测试，并且
能清楚说明局限性。

## 作品集标准

每个成熟项目都应包含：

1. 清楚的问题和目标用户；
2. 数据来源、许可和局限；
3. 可复现的运行方式与依赖；
4. 基线方案与评估指标；
5. 结果、错误案例和下一步；
6. 简短的架构图或流程图；
7. 不提交密钥、隐私数据或大模型权重。

建议的项目序列：

1. 课程数据分析项目：CSV → EDA → 统计/基线模型 → 报告；
2. 数据库 + Python 项目：PostgreSQL → SQL → Pandas → 可视化/API；
3. 端到端 ML 项目：特征、交叉验证、误差分析、简单部署；
4. LLM/RAG 项目：可靠检索、带来源回答、评测集、失败案例与成本说明。

## 原计划的明确调整

- **保留**：Python → 数据处理 → SQL → ML → PyTorch 的主顺序。
- **提前并行**：数学、测试、调试、文档和软件工程习惯，从现在开始少量持续学习。
- **减少刷题配额**：LeetCode 不是每周硬性 5 题；先在学会列表/字典后每周做
  1–2 道并真正复盘。课程和项目优先。
- **不把 Kaggle 分数当目标**：优先真实问题、可复现流程、指标与错误分析。
- **不等待所有阶段完成才接触 PyTorch**：完成一个基础 ML 项目后即可开始 PyTorch
  入门，同时继续补数学。
- **新增 LLM 系统能力**：RAG、评测、工具调用、安全、成本与延迟，而不只学
  Transformer 和 BERT 微调。
- **新增交付能力**：测试、Docker、CI、API/部署和监控；GitHub 不是只放代码，
  而是展示工程判断。
- **取消硬件焦虑**：M2 + 云端 GPU 能覆盖当前学习目标。只有真实项目反复受限时，
  再根据工作负载决定硬件。

| 之后 | 选择研究工程、ML 平台或应用 AI 方向，深耕一个领域并持续复现/贡献开源 |

## 学习原则

1. 每次只学一个小概念，但每周必须把概念放回一个真实问题。
2. 先自己写出第一版；卡住后看提示并解释改动原因。
3. 每个项目先有可运行基线，再提高性能或加新框架。
4. 成绩、诚信、数据安全和可解释性不能为了“炫技”让步。
5. 通过持续交付和反思来积累能力，不追逐短暂的工具热度。
